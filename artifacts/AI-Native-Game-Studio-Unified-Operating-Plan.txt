1. Initial AI Studio Onboarding & Repository Request
To scale game development in the age of generative models, we must solve the "reasoning cliff." While modern models boast massive context windows, their logic begins to liquefy as conversations approach the 150k–200k token threshold. To prevent this decay, we must transition from "Chat-as-Memory" to "Repo-as-Memory." By shifting the project's source of truth from volatile chat threads to persistent Markdown files within the codebase, we turn "AI memory" from transient fog into paperwork with teeth. This ensures that every new chat session starts with a crystalline understanding of the project's laws and state.
Use the following prompt to initialize your AI assistants (the "AI Friends") across all three repositories:

2. The Unified AI-Native Workflow Strategy
The studio operates on the "Architect-Builder-Reviewer" triad. We do not permit multiple models to architect the same ticket; that is how you end up with "three brilliant raccoons fighting over the same sandwich." By assigning ChatGPT 5.4 Pro as the Architect (Director), Codex 5.4 as the Builder (Engineer), and Claude Code as the Reviewer (QA), we maintain clear boundaries and prevent the chaos of conflicting logic.
The Golden Development Loop: Repo-as-Memory
The "Golden Development Loop" mandates that we start a fresh chat for every major task. We rely on the repository itself to carry context across these threads. This prevents "Reasoning Decay"—the quality loss seen in threads over 150k tokens—by ensuring the model always has a clean slate but a detailed set of instructions in the Markdown memory files.

3. Strategic Execution Plans by Game
We are prioritizing a "Workflow Restart" over a "Codebase Restart." Our existing games have validated progress; we are not nuking the foundations, but rather moving the furniture into a new, more stable house with disciplined AI management.
Mr. Football Dynasty (MFD): Monolith Discipline
The strategy for MFD is "Monolith Discipline." We declare game.js as the canonical authored source. We must maintain strict parse/hash discipline to ensure parity across any mirrored JS copies. Prioritize deterministic simulation and franchise depth (rosters, contracts, drafts) while protecting the SAVE_VERSION stability. Do not attempt to split the monolith into multiple files at this stage; work within the established seam.
Mr. Baseball Dynasty (MBD): Typed Architecture
MBD is our model for professional engineering, utilizing a "Typed Architecture" and "worker-owned truth." The focus is on statistical depth and historical immersion through "proof-window tuning"—calibrating signals and thresholds to ensure the simulation engine produces realistic data. Protect the canonical HOME snapshot truth and avoid introducing "component logic soup."
Cactus Ed’s Happiest Place (CEHP): Certification-First
CEHP is in "Certification-First" mode; architecture rewrites are explicitly forbidden. The bottleneck is not code, but human evidence. No pass shall occur without literal human hand-play evidence from W2/W3. We use manual certification notes to guide Codex in patching physics and collision logic defects only. Human playtest evidence is the primary source of truth over automation.
These specific mandates flow directly into the implementation prompts in the following sections.

6. The 7-Day Studio Activation Roadmap
This stabilization week installs the workflow without disrupting current momentum. We are moving from "writing code" to "building a system that produces code."
* Day 1: Canonical Clone Selection. Identify one true local worktree/clone for MFD, MBD, and CEHP.
* Day 2: Fresh Task Chats. Stop using long-lived threads. Initiate fresh implementation chats for the first set of tickets.
* Day 3: Rule Authoring. Create the AGENTS.md for each repo, defining the permanent "laws" (e.g., "UI cannot modify engine state").
* Day 4: State Modeling. Define the GameState model (e.g., GameState.ts) to represent the entire world for serialization/saving.
* Day 5: Verification Setup. Build and test verification scripts (npm run test, npm run build) for agents to run.
* Day 6: First Loop Execution. Execute a single task pass (Architect -> Builder -> Reviewer) for each game.
* Day 7: Architecture Stability Review. Audit the HANDOFF.md and AGENTS.md files for drift and reasoning quality.
The Most Important Mindset: The objective is not to write code faster, but to build a repeatable system that produces professional results. By treating the repository as the memory and the AI as disciplined roles in a triad, you are building an AI-native game studio capable of long-term, high-quality production.
