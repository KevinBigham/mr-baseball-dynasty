⸻

The Unified Architecture for Your Game Studio

All three games should share the same top-level philosophy.

Think of the project as three stacked layers:

Engine → Systems → Presentation

The engine is the truth.
Systems manipulate the truth.
Presentation displays the truth.

This separation is the secret behind long-lasting games.

⸻

The Studio-Level File Structure

Across all three games, your repos should look roughly like this:

/game-name
  /engine
  /systems
  /ui
  /data
  /assets
  /tests
  /tools
  AGENTS.md
  TASK.md
  HANDOFF.md

These folders give AI agents extremely clear boundaries.

Each folder answers a different question.

Engine: What are the rules of the game?

Systems: How do features interact with those rules?

UI: How does the player see and interact with the game?

Data: What configuration drives the simulation?

Assets: Art, sounds, fonts

Tests: Verification

Tools: Developer automation

Game 1: Mr Football Dynasty

This game is a league simulation engine.

The most important thing here is deterministic simulation.

That means if the engine runs twice with the same seed, it produces the same results.

That’s what allows:

• stable save files
• reproducible bugs
• historical league simulation
• future portability

Engine Modules

/engine
  leagueEngine.ts
  gameSimEngine.ts
  rosterEngine.ts
  contractEngine.ts
  progressionEngine.ts
  rng.ts

These files contain pure logic only.

No UI code.

No React.

No browser APIs.

Just math and state transitions.

Examples:

gameSimEngine → simulates a football game
rosterEngine → roster rules and limits
contractEngine → salary cap and contracts
progressionEngine → player development

⸻

Systems Layer

/systems
  draftSystem.ts
  tradeSystem.ts
  scoutingSystem.ts
  scheduleSystem.ts
  injurySystem.ts
  aiGMSystem.ts
  saveSystem.ts

These systems call the engine but don’t define the rules.

Example:

The draft system calls rosterEngine functions to place players.

The AI GM system calls tradeSystem logic.

The save system serializes the engine state.

⸻

UI Layer

/ui
  dashboard
  roster
  trade
  draft
  league
  frontOffice

These are your React components.

They read from systems and display information.

Important rule:

UI never mutates engine state directly.

The Most Important Concept

Every game should maintain a single world state object.

Example:

GameState

Inside this object lives the entire universe.

Players
Teams
Stats
Scores
Level state

Systems update GameState.

UI reads GameState.

The engine transforms GameState.

This makes saving and loading trivial.

Just serialize GameState.

⸻

Why This Architecture Works for AI

This structure dramatically improves AI collaboration.

AI agents no longer guess where things belong.

They know exactly where to work.

Engine work stays in engine.

UI work stays in UI.

System work stays in systems.

That alone cuts a huge amount of AI confusion.
